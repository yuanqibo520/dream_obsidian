## 3. 目标目录结构

根据实际项目补充目录结构。AI 编程时应遵守本结构，不要随意新建重复目录。

### 前端示例

```txt
src/
  app/
  components/
    common/
    layout/
    feature/
  pages/
  hooks/
  lib/
  services/
  stores/
  types/
  styles/
```

### 后端示例

```txt
app/
  api/
    routes/
  core/
  models/
  schemas/
  services/
  repositories/
  db/
  utils/
tests/
```

### 目录职责

- `components/common`：通用 UI 组件。
- `components/layout`：布局组件。
- `components/feature`：业务组件。
- `services`：接口请求与外部服务封装。
- `stores`：全局状态。
- `types`：共享类型定义。
- `lib`：通用工具函数。
- `api/routes`：后端路由层，只处理请求和响应。
- `services`：后端业务逻辑层。
- `repositories`：数据库访问层。
- `schemas`：请求和响应数据结构。

## 4. 开发规范

### 通用规范

- 修改代码前，先阅读相关文件，理解现有结构和风格。
- 优先沿用项目已有写法，不随意引入新的架构。
- 不做无关重构。
- 不删除用户已有逻辑，除非任务明确要求。
- 不擅自更换技术栈。
- 不新增大型依赖，除非说明原因并获得确认。
- 每次改动后说明修改了哪些文件、为什么修改。
- 能运行测试或构建时，改完后必须运行验证命令。

### 前端规范

- 优先使用 TypeScript。
- 不使用 `any`，除非有明确理由。
- API 请求统一放在 `services/`。
- 组件不直接写复杂请求逻辑和业务逻辑。
- 页面必须考虑 `loading`、`error`、`empty` 状态。
- 表单必须处理提交中、提交失败、校验失败状态。
- 移动端不能出现横向溢出。
- UI 风格应与现有组件保持一致。

### 后端规范

- 路由层只处理请求参数、权限校验入口和响应返回。
- 业务逻辑放在 `services/`。
- 数据库操作放在 `repositories/` 或项目既有数据访问层。
- 请求和响应结构使用 schema 定义。
- 错误返回格式保持统一。
- 新增核心逻辑时补充测试。

## 6. 验收标准

每个任务开始前尽量写清楚验收标准。

常见验收标准示例：

- 构建命令通过。
- 测试命令通过。
- 空数据状态正常显示。
- 请求失败状态正常显示。
- 表单校验正常工作。
- 移动端布局不溢出。
- 没有修改无关文件。
- 没有引入未确认的新依赖。

## 7. 推荐工作流

```txt
1. 写清楚需求
2. 让 AI 阅读 AI_RULES.md 和相关代码
3. 让 AI 拆解实现方案
4. 按小任务逐步实现
5. 每步后运行测试或构建
6. 让 AI 总结修改内容
7. 人类做最终验收
```

## 8. 禁止事项

- 禁止在未确认的情况下重写整个项目。
- 禁止把所有逻辑写进一个大文件。
- 禁止重复创建相同职责的目录。
- 禁止忽略已有组件和工具函数。
- 禁止只实现正常状态，不处理异常状态。
- 禁止为了修复小问题而引入复杂架构。
