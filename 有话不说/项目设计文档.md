# 英语单词记忆网站项目设计文档

## 一、项目背景与目标

本项目旨在帮助用户通过输入遇到的生词，建立个人专属单词库，并通过英语释义、趣味例句（优先贴近真实刊物语体）和标准美式发音，提升记忆效率和实际应用能力。

**视觉与信息架构采用「设计系统方案 B」**：主学习流采用浅色玻璃拟态（Glassmorphism）以突出沉浸与轻盈感；打卡、统计与日历类高密度信息区采用深色卡片（参考主流「每日任务」日历的信息结构），但与全站共享圆角尺度、间距体系与字体家族，避免两套产品拼贴感。

---

## 二、设计系统层级（方案 B，已锁定）

| 层级 | 适用界面 | 视觉关键词 | 与全站关系 |
|------|-----------|-------------|------------|
| L1 主学习流 | 单词录入、释义、发音、复习主流程 | 浅色渐变背景、`backdrop-filter` 模糊、半透明白玻璃面板、紫色主强调色、大圆角（约 24–32px）、轻阴影 | 默认用户停留时间最长的「品牌层」 |
| L2 打卡与统计 | 每日背诵打卡、月历、连续天数、掌握量折线图 | 深色圆角容器（约 12–16px）、高对比度字色、绿色表示今日/达标、橙色表示已打卡点或挑战高亮 | 嵌入在 L1 页面中的「信息密度层」，外框仍可为玻璃卡片包裹 |
| L3 导航与框架 | 顶栏、底栏 | 与 L1 一致的玻璃/线性图标；选中态使用紫色实心衬底 | 全局一致 |

**说明**：早期文档中的「黏土拟态」不作为当前阶段目标；若后续单独做营销页，可再评估是否复用。

## 三、核心功能

1. **单词录入**：用户可随时输入不会的英语单词，自动存入个人单词库。
2. **释义展示**：为每个单词提供权威的英语释义（来源优先级与冲突处理见下文「功能边界」）。
3. **趣味例句生成（有边界）**：为每个单词生成辅助记忆的短例句；**体裁优先级**为：**赞美**、**问候**、**提问**、**谚语式/格言式**表达；语料风格可对齐严肃刊物句式（见「例句语料与合规」）。
4. **标准美式发音**：支持单词的标准美式发音播放。
5. **个人单词库管理**：支持单词的增删查改，便于复习和管理。
6. **英语每日背诵打卡**：用户**自行选择所在地时区**；打卡与时长统计**不引入「暂停」状态**与**「最小有效时长」门槛**（即：以一次学习会话的起止时间或简单累计计时为准，产品规则保持简单，后续若需防刷再迭代）。
7. **单词掌握量进步折线图**：在统计区展示随时间变化的掌握量（或「已复习词条数」等可量化指标），用于反馈长期进步。

## 四、功能边界与验收（缓解范围模糊）

### 4.1 趣味例句

- **必须满足**：例句为完整英文句子；长度建议上限约 280 字符；生词目标词在句中自然出现 1 次为宜。
- **体裁优先级（生成策略）**：① 赞美 ② 问候 ③ 提问 ④ 谚语/格言式短句；可在 Prompt 或规则引擎中写死权重。
- **禁止或慎用**：未经审核的敏感话题、人身攻击、虚假新闻体；涉及真人媒体引述时需有合法来源支撑。
- **释义冲突**：同一单词多来源时，默认展示「词典权威义项」为主，刊物例句仅作风格参考，不替代词典定义。

### 4.2 市面可参考的英语文学/人文刊物（例句风格对齐用）

以下刊物以「语言正式、文学性强或深度报道」著称，可作为**风格与句式**参考方向（实际入库须遵守版权与条款）：

| 刊物 | 说明（简述） |
|------|----------------|
| *The Paris Review* | 文学访谈与小说节选，文学性强 |
| *Granta* | 虚构与非虚构并重，叙事密度高 |
| *The New Yorker* | 深度报道与文学版块兼顾，通用性强 |
| *Times Literary Supplement*（TLS） | 书评与文化评论，学术与文学交界 |
| *London Review of Books* | 长文评论，句式复杂度高 |
| *Harper's Magazine* | 美国人文与政经长文 |
| *Poetry*（Poetry Foundation） | 诗歌文本，适合节奏与凝练表达参考 |
| *The Atlantic* | 文化、科技与社会议题长文 |

**若上述来源在授权与工程上难以统一落地**：**优先选用《纽约时报》（The New York Times）公开可查文本**作为语料主轴之一，建立**语料库表**存储「近年来已合法公开获取」的篇章或句子级索引，供例句检索、搭配统计与风格分析（见下一节）。

## 五、例句语料与数据库（NYT 优先策略）

### 5.1 目标

建立可查询的语料存储，用于：高频搭配、典型句式、难度分层与例句生成模型的检索增强（RAG）或离线分析。

### 5.2 合规与来源（必读）

- **不得**在文档层面鼓励违反服务条款的大规模抓取付费内容。
- **应优先**：官方/合作方提供的 **API**、**授权数据集**、**已进入公有领域（Public Domain）** 的文本，或机构提供的**研究用许可语料**。
- 对 NYT：需查阅 [The New York Times Developer 文档与条款](https://developer.nytimes.com/)，仅使用条款允许范围内的接口与字段；付费墙内正文默认**不入库**。

### 5.3 库表设计意向（字段级留档，便于后端对齐）

- **语料篇级表**：`article_id`、`source`（如 NYT）、`title`、`published_at`、`url`、`language`、`license_tier`（公开/ API / 授权）。
- **语料句级表**：`sentence_id`、`article_id`、`text`、`tokenized`（可选）、`created_at`。
- **索引**：全文检索（PostgreSQL GIN / 专用搜索引擎）与按日期、来源过滤。

### 5.4 与用户例句的关系

用户侧「趣味例句」可优先从语料库检索含该词的句子片段，经改写/缩短后输出；若无命中再调用大模型，并记录 `source_attribution` 为 `corpus` / `llm` / `template`。

## 六、用户体系与数据模型（当前 Web，预留多端升级）

### 6.1 当前范围

- **仅支持网站发布**；多端同步、原生 App 等列入后续升级，不在 v1 必交付范围。

### 6.2 最小用户模型

- **唯一用户 ID**（`user_id`，UUID 或雪花 ID，全局主键）。
- **账号**（登录名或邮箱，唯一约束）。
- **密码**（仅存储加盐哈希，如 Argon2id / bcrypt）。
- **时区**：用户自选 IANA 时区字符串（如 `Asia/Shanghai`），用于「自然日」划分与打卡日历。
- **角色**：`normal` | `developer`（开发者账号用于联调、灰度功能、详细日志开关；生产环境需 IP 白名单或二次验证，见非功能需求）。

### 6.3 开发者账号

- 目的：便于后续优化与 **debug**（可查看扩展日志、复现用户问题时不触碰真实用户数据的原则下使用）。
- 要求：与正式数据隔离策略（单独 `environment` 或单独库）、强审计日志。

### 6.4 预留多端同步升级路径（设计层）

- 鉴权从「纯 JWT」演进为「短效 Access Token + 长效 Refresh Token + 设备表」。
- 增加 `devices` 表与同步版本向量或 `updated_at` 乐观锁；冲突策略（最后写入优先 / 用户合并）后续产品定稿。

### 6.5 核心实体关系（概要）

- `users` 1—N `vocabulary_items`
- `users` 1—N `checkin_sessions`（一次打开学习到结束的记录，**无暂停字段**）
- `users` 1—N `daily_checkin`（按用户时区日期聚合：是否打卡、总时长等）
- `vocabulary_items` 可关联 `mastery_events` 供折线图聚合

## 七、每日打卡与统计（与时区、图表）

### 7.1 规则（按当前产品决策）

- **时区**：用户自行选择所在地时区；所有「日」边界按该时区 0 点划分。
- **计时**：不设计「暂停」；不设「最小有效时间」；以会话记录的开始/结束时间差或前端简单累计为准（实现简单，接受一定「挂机」风险，后续可加页面失焦衰减等增强）。
- **打卡判定**：当日存在至少一条有效学习会话且达到产品定义的「完成」条件即可（如：会话正常结束；具体布尔规则在接口文档中写死）。

### 7.2 单词掌握量进步折线图

- **横轴**：日期（按用户时区）。
- **纵轴**：可选指标之一——「当日新增掌握词条数」累计、「掌握度加权分」等（指标公式在数据分析章节单独立项，避免与词典义项混淆）。
- **展示位置**：L2 深色统计区，与月历同页或相邻 Tab。

## 八、技术方案（默认路线，降低决策债务）

### 8.1 前端（默认）

- **默认工程路线：Next.js（基于 React）**。理由：统一 React 生态、路由与构建工具链成熟、后续若需 SEO 或服务端能力可渐进启用。
- **备选**：若明确不需要 SSR/SSG，可采用 **React + Vite** 单页应用；团队需在立项时二选一写入里程碑，**不再长期并列「或」而不决**。
- UI：按方案 B 自定义 CSS/变量 + 少量组件库二次封装；玻璃拟态依赖 `backdrop-filter` 与降级策略（不支持时降低模糊强度）。

### 8.2 后端（默认）

- **Node.js + Express**（默认）；复杂中间件链再评估 Nest。
- **数据库默认：PostgreSQL**（用户、语料索引、打卡、掌握事件等关系型数据清晰）；文档型需求可用 JSONB 字段兼容。

### 8.3 第三方服务

- 词典 API、TTS API、大模型 API（例句生成）；NYT 等语料仅走合法接口或许可数据。

### 8.4 部署

- 前端：Vercel（与 Next 契合）或 Netlify、自有服务器。
- 后端：云服务器或 Docker；数据库托管实例；开启自动备份。

### 8.5 接口规范

- OpenAPI/Swagger 为单一事实来源；前后端 Mock 基于同一契约。

## 九、非功能需求与市面常用方案（对比）

> 本节给出常见工程方案及优缺点，便于选型时在评审表中勾选。

### 9.1 身份与鉴权

| 方案 | 优点 | 缺点 |
|------|------|------|
| **JWT（存 LocalStorage）** | 无状态、易水平扩展 | XSS 时令牌易被盗；需严格 CSP 与输入消毒 |
| **Session + HttpOnly Cookie** | 降低 XSS 窃取风险 | 需服务端存储或 Redis；跨域与集群粘性要设计 |
| **JWT + HttpOnly Cookie（双持）** | 兼顾无状态与安全倾向 | 实现与刷新逻辑略复杂 |

**建议默认**：生产环境倾向 **HttpOnly Cookie 承载 Refresh + 短效 Access**（或 BFF 转发），配合 HTTPS。

### 9.2 限流与防滥用

| 方案 | 优点 | 缺点 |
|------|------|------|
| **应用内中间件限流**（如 `rate-limiter-flexible` + Redis） | 成本低、可控细 | 需自建 Redis 与高可用 |
| **API Gateway / CDN 限流**（如云厂商、Cloudflare） | 挡在边缘、抗 DDoS 好 | 规则粒度略粗；费用随流量 |
| **全局限流 + 用户级配额** | 防单用户刷爆 LLM/TTS | 配置与监控工作量大 |

### 9.3 日志与可观测性

| 方案 | 优点 | 缺点 |
|------|------|------|
| **结构化日志 + 云厂商日志服务** | 上手快、与主机绑定少 | 厂商锁定、费用随存储增长 |
| **ELK / OpenSearch** | 查询灵活、生态成熟 | 自建运维成本高 |
| **Grafana Loki + Prometheus** | 指标与日志统一体验好 | 学习曲线与组件多 |

### 9.4 错误监控

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Sentry** | 前后端统一、Release 跟踪强 | SaaS 费用；国内访问需评估 |
| **自建 GlitchTip 等** | 可控数据驻留 | 维护成本 |

### 9.5 密钥与配置

| 方案 | 优点 | 缺点 |
|------|------|------|
| **环境变量 + 启动注入** | 简单 | 易泄露在镜像层；轮换不便 |
| **云 Secret Manager / Vault** | 轮换、审计、细粒度授权 | 配置与成本 |

### 9.6 数据库备份与恢复

| 方案 | 优点 | 缺点 |
|------|------|------|
| **云托管自动快照 + PITR** | 运维省心、RPO 低 | 费用随保留周期升 |
| **逻辑备份（pg_dump）到对象存储** | 便宜、可异地 | RPO 取决于调度频率 |

### 9.7 HTTPS 与边缘安全

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Let's Encrypt / 云托管证书** | 标准化、自动化 | 需续期监控 |
| **WAF + CDN**（如 Cloudflare） | 抗扫、抗 CC | 规则误杀需调优 |

### 9.8 密码存储

- **Argon2id** 或 **bcrypt**；禁止明文或可逆加密。

## 十、可能遇到的问题与建议（修订）

1. **接口**：OpenAPI 契约 + Mock；变更走版本号。
2. **释义/TTS/LLM 成本**：配额、缓存、降级（仅释义无例句）。
3. **语料版权**：仅合法渠道；语料表记录 `license_tier` 便于审计。
4. **UI 降级**：`backdrop-filter` 不支持时降级为实色浅底 + 轻边框。
5. **挂机刷时长**：当前版本接受简单规则；若恶化再引入失焦折扣或心率式验证（非现阶段）。
6. **性能**：音频 CDN；数据库索引；热点词缓存。

---

## 十一、总结

本项目采用前后端分离架构，**设计系统明确为方案 B（玻璃主学习流 + 深色打卡统计区）**；**前端默认路线为 Next.js（React）**；用户模型以 **账号 + 密码 + 唯一 ID** 为主并设 **开发者账号**；例句生成有明确体裁优先级与语料库（**优先 NYT 合法公开数据**）支撑；打卡按**自选时区**、**无暂停与最小时长**；统计侧提供**掌握量折线图**。开发前期应完成：接口契约、语料合法来源、鉴权与密钥方案、数据库备份策略；后期再演进到多端同步与更强防滥用能力。
